# Phase 1: Core Foundations - Context

**Gathered:** 2026-03-04

<domain>
## Phase Boundary

Externalize hardcoded orca layer definitions (ROOT_GROUP_NAME, LAYER_STRUCTURE, LAYER_COLORS, LAYER_SECTIONS) from `spectrace_annotator.py` into YAML template config files. After this phase, changing a template file changes the layers that get created — no code edits required. The existing orca annotation workflow must still work end-to-end.

</domain>

<decisions>
## Implementation Decisions

### Config file format
- YAML format for all template files — consistent with existing orca_template.yaml convention
- Templates include inline comments explaining each layer's scientific purpose (self-documenting for biologists)
- Colors merged into the YAML template alongside layer structure — one file = one species definition
- Hex color strings (#FF0000) for readability; code converts to RGB tuples as needed

### Template structure
- Nested YAML hierarchy — groups contain their children as nested keys, mirroring the GIMP layer tree
- All-in-one template: spectrogram params + layer hierarchy + colors + UI sections in a single file
- UI sections (LAYER_SECTIONS) included in template — full species customization, not derived from hierarchy
- Version field included — stored as GIMP parasite for template-change detection by export pipeline

### Template discovery
- Templates live in `templates/` directory, shipped with spectrace
- Plugin functions accept template name as parameter (default: 'orca') — plumbing for species selection exists from Phase 1
- Auto-discovery: any YAML file in templates/ with the right structure is a valid template
- Malformed templates fail loud with clear error messages — no silent fallback, consistent with opinionated UX philosophy

### Migration path
- YAML replaces XCF as the single source of truth for layer definitions
- Rewrite existing `orca_template.yaml` into the new machine-readable format (same path, same git history, descriptions become inline comments)
- Full refactor — all consumers (GIMP plugin, utils.py, hdf5_utils.py) read from YAML template. One source of truth, no split behavior.
- Shared Python module (`spectrace/core/templates.py`) handles all template loading/validation — used by both GIMP plugin and CLI tools. Starts the `spectrace/core/` directory.

### Claude's Discretion
- Exact YAML schema design (key names, nesting depth)
- Template validation implementation details
- How to handle the `layer_color_mapping.json` migration (may become unnecessary)
- Whether to keep `orca_template.xcf` as a visual reference or remove it

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `templates/orca_template.yaml`: Documentation-only YAML with layer descriptions — rewrite into machine-readable format
- `templates/orca_template.xcf`: Binary GIMP template — superseded by YAML but may stay as visual reference
- `utils.py:create_color_mapping()`: Auto-generates colors for layers — may adapt to read from YAML colors
- `utils.py:get_or_create_color_mapping()`: Load/create color mapping — refactor to read from template
- `class_registry.py`: Append-only class versioning — continues to work, reads class names from template

### Established Patterns
- Path-qualified layer names (e.g., "Heterodynes/5") used throughout codebase — must preserve this convention
- `layer_group_name` parameter already exists in many utils functions (default "OrcinusOrca_FrequencyContours") — refactor to read from template
- `gimpformats.GimpDocument` used for XCF parsing — may be reduced/removed as YAML becomes source of truth
- Color mapping JSON (`projects/layer_color_mapping.json`) generated by utils — may become unnecessary if colors live in template

### Integration Points
- `spectrace_annotator.py:create_template_layers()` — primary injection point, reads from hardcoded constants
- `spectrace_annotator.py:build_layer_index()` — maps layer paths to GIMP objects
- `spectrace_annotator.py:SpectraceBackgroundMonitor` — reads LAYER_COLORS for auto-color switching
- `utils.py:extract_layers_from_xcf()` — reads XCF for layer names, needs to work with YAML-defined structure
- `utils.py:discover_layer_names_from_template()` — currently reads from XCF template
- `hdf5_utils.py` — validates project XCFs against template layer list, creates HDF5 datasets

</code_context>
